# 분양현장 지도 맵핑 사이트 — 구현 계획서

> 작성일: 2026-05-30 · 대상: `prus.kr/portal` 내 신규 모듈
> 목적: 분양 설문 데이터를 기반으로 전국 분양현장을 지도에 맵핑하고, GIS·현장 대시보드와 연동하는 사이트 구현

---

## 1. 한 줄 요약

보유 중인 현장 마스터 CSV(5,749건)를 정제·지오코딩하여 **지도 위에 현장을 시각화**하고, 마커 클릭 시 **현장별 설문/분양 정보 상세**를 보여주며, **지역·브랜드·기간 단위 대시보드**로 현황을 집계하는 단일 웹 모듈을 만든다.

## 2. 데이터 현황 (실제 CSV 분석 결과)

분석 대상: `data/피알어스_현장_202605301913.csv`

| 항목 | 내용 |
|---|---|
| 총 행 수 | 5,749건 (숨김 `Y` 245건 제외 시 5,504건) |
| 컬럼 수 | 34개 |
| 위경도 좌표 | **없음** → 지오코딩 필요 |
| 핵심 위치 필드 | `지역`(채움 5,702건, 예: "경기도 평택시"), `위치`(채움 5,473건, 주소 텍스트) |
| 분류 필드 | `브랜드명`, `시공사`, `시행사`, `분양대행사` |
| 기간 필드 | `시작일자`, `마감일자` (설문 진행 기간) |
| 규모 | `규모`(텍스트, 예: "총 1,941세대 공급") → 숫자 파싱 필요 |
| 설문 콘텐츠 | `설문인사말`(2,620건), `응답코드`/`결과값`(가상번호 발급 로그) |
| 운영 메타 | `안심번호`, `부재중안심번호`, 마트/점포명, 담당자 등 |

### 지오코딩 사전 검증 결과 (중요)

좌표가 없으므로 `지역`+`위치` 텍스트를 행정구역 중심좌표 사전과 매칭하는 폴백 방식을 시험한 결과:

- **시·군·구 단위 이상 매칭: 95.4% (5,482건)**
- 시·도 단위 개략 매칭: 177건 / 매칭 실패: 90건
- 신도시·택지지구 명칭(동탄→화성, 광교→수원, 송도→연수구 등) 별칭 처리로 1,157건 추가 정밀화

→ **결론: API 없이도 구·동 수준 근사 맵핑은 즉시 가능. 단지 단위 정밀 핀은 외부 지오코딩 API 필요.**

## 3. 지오코딩 방안 (핵심 의사결정)

정밀도와 비용의 트레이드오프. 3단계 전략 권장:

1. **(즉시) 행정구역 중심 폴백** — 사전 기반, 무료, 95% 커버. MVP·프로토타입용.
2. **(권장) 외부 지오코딩 API 1회 배치** — `위치` 주소 텍스트를 좌표로 변환해 DB에 캐싱.
   - 후보: **카카오 로컬 API**(주소→좌표, 한국 주소 정확도 높음, 무료 쿼터 충분), **VWorld**(정부 공공, 도로명/지번 강점), 네이버 클라우드 Geocoding.
   - 5,749건 1회 변환 후 좌표 컬럼 저장 → 이후 신규 현장만 증분 변환.
3. **(보완) 실패분 수기 보정** — API도 실패하는 비정형 주소(예: "C5,C8블럭")는 관리자 화면에서 핀 드래그 보정.

> 권장 순서: MVP는 1번으로 빠르게, 정식 버전은 2번으로 좌표를 영구 저장하고 1번을 폴백으로 유지.

## 4. 시스템 아키텍처

```
[현장 CSV / DB]
      │  (1) ETL: 정제·파싱·지오코딩
      ▼
[정규화 데이터 스토어]  ──  sites 테이블 (id, name, region, sido, brand,
      │                      lat, lng, geolevel, units, start, end, status, ...)
      │
      ├─(2) REST/GraphQL API ──►  [지도 프론트엔드]  ─ 마커 클러스터 · 필터 · 상세
      │                                   │
      ├─────────────────────────►  [대시보드]  ─ 지역/브랜드/기간 집계
      │
      └─(3) 설문 응답 연동  ──►  현장별 설문 통계 (응답수, 선호평형, 관심연령 등)
```

연동 대상(요청사항): **① 지도/GIS, ② 현장 대시보드.**

---

## 5. 데이터 파이프라인 (ETL)

CSV → 서비스 데이터로 변환하는 전처리 단계. (검증 완료된 로직)

1. **정제**: 결측 처리, `숨기기=Y` 필터, 중복 현장 정리
2. **위치 파싱**: `지역`/`위치`에서 시·도 + 시·군·구 추출, 신도시 별칭 매핑
3. **지오코딩**: 위 3단계 전략으로 `lat`, `lng`, `geolevel`(정밀도 등급) 부여
4. **세대수 파싱**: `규모` 텍스트 → 정수 `units` (정규식)
5. **상태 산출**: `마감일자` vs 오늘 기준으로 `진행중/예정/종료/미정`
6. **적재**: DB 테이블 또는 정적 JSON으로 출력 (현장 수 6천건 규모는 정적 파일로도 충분)

## 6. 기능 정의

### 6.1 지도 화면 (MVP 필수)

- 전국 지도 + **현장 마커**(상태별 색상: 진행중/예정/미정/종료)
- **마커 클러스터링** (6천건 → 줌 레벨별 군집화 필수)
- **필터**: 시·도, 분양 시작연도, 상태, 브랜드/현장명 검색
- **마커 클릭 → 상세 패널**: 브랜드·시공/시행사·규모·세대수·분양기간·담당자·연락처·설문인사말
- 좌표 정밀도 뱃지(단지/구/시군구/시도) 표기

### 6.2 대시보드 (연동 대상)

- KPI: 표시 현장 수, 진행/예정 현장 수, 총 세대수
- **지역별 분포** 막대 (상위 N)
- **브랜드별 분포** 막대
- 연도별 분양 추이 (선택)
- 필터와 실시간 연동 (지도와 동일 필터 상태 공유)

### 6.3 GIS 연동 (확장)

- 행정구역 경계(시군구 폴리곤) 위에 현장 밀도 **코로플레스(choropleth)** 표시
- 입지 분석: 반경 내 현장 수, 경쟁 현장 클러스터

### 6.4 설문 연동 (확장)

- 현장 상세에 **설문 응답 요약**(응답 수, 선호 평형, 관심 연령대, 가격 민감도) 차트
- `응답코드`/안심번호 로그를 응답 테이블과 조인

### 6.5 관리자 (운영)

- 신규 현장 등록 시 자동 지오코딩 + 핀 수동 보정
- 숨김/노출 토글, 데이터 재적재 트리거

## 7. 기술 스택 제안

> `prus.kr/portal` 기존 스택에 맞추는 것을 우선. 아래는 일반 권장안.

| 영역 | 권장 | 비고 |
|---|---|---|
| 지도 라이브러리 | **카카오맵 SDK** 또는 Leaflet | 한국 서비스는 카카오맵이 주소/로드뷰 강점. 오픈소스 선호 시 Leaflet+OSM/Carto |
| 마커 군집 | 카카오 Clusterer / Leaflet.markercluster | 6천건 필수 |
| 프론트 | 기존 portal 프레임워크 재사용 | React/Vue 등 |
| 차트 | Chart.js 또는 Recharts | 대시보드 막대/추이 |
| 백엔드 | 기존 portal API에 `sites` 엔드포인트 추가 | 필터 쿼리 파라미터 지원 |
| 데이터 | RDB `sites` 테이블 + 좌표 캐시 | 6천건은 정적 JSON 캐싱도 가능 |
| 지오코딩 | 카카오 로컬 API / VWorld | 1회 배치 + 증분 |

## 8. 단계별 로드맵

| 단계 | 산출물 | 핵심 작업 |
|---|---|---|
| **0. 데이터 정합성** | 정제 리포트 | 컬럼 정의 확정, 결측·중복·상태 규칙 합의 |
| **1. MVP 지도** | 동작하는 지도 화면 | ETL(폴백 지오코딩) → 마커·클러스터·필터·상세 |
| **2. 대시보드** | 집계 화면 | KPI·지역/브랜드 분포, 필터 연동 |
| **3. 정밀 지오코딩** | 좌표 영구화 | 카카오/VWorld 배치 변환 + 관리자 핀 보정 |
| **4. 설문 연동** | 현장별 설문 통계 | 응답 데이터 조인, 상세에 차트 |
| **5. GIS 고도화** | 밀도·입지 분석 | 행정경계 코로플레스, 경쟁 분석 |

MVP(1단계)는 이미 검증된 ETL 로직이 있어 빠르게 도달 가능.

## 9. 리스크 및 고려사항

- **좌표 정밀도**: 폴백 방식은 구/동 수준 근사라 같은 구의 현장이 겹쳐 보일 수 있음 → 클러스터링 + 정식 지오코딩으로 해소.
- **주소 비정형성**: "C5,C8블럭" 같은 블록 단위 주소는 API도 실패 가능 → 수기 보정 플로우 필수.
- **개인정보**: `안심번호`, `연결전화번호`, 담당자 연락처 노출 범위 권한 설계 필요(내부용/외부용 분리).
- **데이터 신선도**: 현장은 계속 추가됨 → CSV 일괄 적재가 아니라 DB 증분/주기 동기화 구조 권장.
- **성능**: 6천건은 정적 JSON으로 충분하나, 수만 건 이상 확장 시 서버사이드 필터링/타일 기반 마커 필요.
